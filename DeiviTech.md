## Learning Philosophy & Workflow

**Learning Approach**
```yaml
"Learn by Doing" Methodology:
  Philosophy:
    - Theory is secondary to practice
    - Real projects teach more than tutorials
    - Mistakes are learning opportunities
    - Iterate, don't perfect on first try
  
  Project-Based Learning Cycle:
    1. Identify Need/Problem:
       Example: "Clients asking for price quotes repeatedly"
       
    2. Research Minimal Viable Solution:
       - Google similar implementations
       - Browse CodePen/GitHub for examples
       - Read documentation (as needed, not comprehensive)
    
    3. Build Messy First Version:
       - Get something working (functionality > beauty)
       - Hardcoded values acceptable
       - Console.log() debugging
    
    4. Test with Real Use:
       - Deploy to GitHub Pages
       - Share with 1-2 clients for feedback
       - Observe actual usage patterns
    
    5. Iterate Based on Feedback:
       - Refactor for readability (now that it works)
       - Extract reusable functions
       - Improve UI/UX based on user comments
    
    6. Document Learnings:
       - README.md with setup instructions
       - Code comments for complex logic
       - Personal notes (what worked, what didn't)
    
    7. Move to Next Challenge:
       - Apply learned patterns to new project
       - Incrementally add complexity
  
  Example: FreelancerDeiviTech Calculator Evolution:
    V1: Static prices in HTML (manual calculation)
    V2: JavaScript with hardcoded prices (basic calculation)

     - Comment out sections to isolate issue
     - Test one function at a time
     - Verify inputs and outputs

     - Paste error + relevant code into Copilot Chat

     - Implement suggested fix, iterate
  
  6. Last Resort - Rewrite:
     - If spent 2+ hours on same bug
     - Sometimes faster to rebuild from scratch
     - "Don't fall for sunk cost fallacy"
